Phase 4 45 min

Feature Development

Objective: Execute the core development loop using the UCI Online Retail dataset. You'll intentionally write complex code, watch it get blocked, then refactor.

1 Data Retrieval

Use the agent to fetch the dataset directly from the workshop repository.

Open Agent Mode (Ctrl+Shift+I) and prompt:

Copy this prompt:
"Download the retail_sample.csv file from https://github.com/Cstewart-HC/programming_w_agents/blob/master/data/retail_sample.csv and save it directly into my local data/ directory."

2 Plan

Open the Copilot Chat Sidebar (Ctrl+Alt+I) and select Plan Mode from the agent selector at the top of the panel. Plan Mode is designed for design-time reasoning - it won't edit files or run terminal commands, but will think through your request and produce a structured plan.

Map out the approach:

Copy this prompt:
"Plan the implementation for src/analysis.py that ingests data/retail_sample.csv. This is the retail sales analytics script we defined in Phase 1. Requirements:
  1. Ingest and validate: Explicitly validate the schema and parse the dates defensively.
  2. Revenue analysis: Calculate total revenue (Quantity * UnitPrice) per Country.
  3. Anomaly detection: Flag any returns (negative quantities) or records with missing CustomerIDs as anomalies.
  4. Console output: Print two clean console tables: a revenue summary by country and an anomaly alert list.
Outline the necessary typed helper functions with docstrings first."
Tip: Take a moment to review the plan before proceeding. Make sure you understand the proposed structure.

3 Build (Intentionally Poor)

Open Copilot Edits (Ctrl+Shift+I). We'll intentionally ask for bad code:

Copy this prompt:
"Implement the plan we just discussed in src/analysis.py. However, do it entirely within a single, highly nested function named process_retail_data ignoring the helper function outline."
Why are we doing this? We want to trigger the pre-commit hook! This simulates what happens when you (or an AI) writes code without considering complexity limits.

4 Review

Stage A: AI Review (Soft Check)

With src/analysis.py active, open the Copilot Chat Sidebar:

Copy this prompt:
"Review this code. Does it adhere to the complexity rules defined in my workspace instructions?"

Copilot may warn you, but AI is fallible. We need a hard gate.

Stage B: Hard Enforcement

In your terminal, run:

pre-commit run --all-files
Expected Result: Ruff will fail with something like: 
C901 `process_retail_data` is too complex (X > 5)
This is the hook doing its job!
No failure? That's okay. Copilot may produce code that already passes the check, even when asked to write a single monolithic function. If the hooks pass, that's still a valid result — it means the AI kept the code simple enough on its own. The important takeaway is that the enforcement mechanism is in place and working. If this happens, you can proceed directly to Step 5 and commit, or re-prompt with a more explicit request for a deeply nested, single-function implementation to try again.

5 Refactor

Return to Copilot Edits and prompt:

Copy this prompt:
"The code failed my pre-commit hook due to high cognitive complexity. Refactor process_retail_data into the smaller, modular helper functions we originally planned so it passes the Ruff linter."

Accept the refactored code. Run pre-commit run --all-files again to verify it passes:

pre-commit run --all-files
Success! Once the hooks pass, use Agent Mode to commit:
Copy this prompt:
"Stage and commit src/analysis.py with the message 'Add retail data analysis script'."

Key Takeaways

  • Plan first, build second - The Plan-Build-Review loop prevents wasted effort
  • AI is a soft check, hooks are hard gates - Trust the automated enforcement
  • Complexity limits force better design - Breaking code into small functions isn't just about passing linting; it's about maintainability
  • Iterate with the AI - Feed error messages back to the agent to get targeted fixes
